# 开发规范

## 代码规范

### Java代码规范
1. 遵循阿里巴巴Java开发手册
2. 使用4个空格缩进，不要使用Tab
3. 类名使用大驼峰命名法：`UserService`
4. 方法名使用小驼峰命名法：`getUserById`
5. 常量使用全大写加下划线：`MAX_COUNT`

### Vue代码规范
1. 组件名使用大驼峰命名法：`UserProfile.vue`
2. 变量名使用小驼峰命名法：`userList`
3. 使用ES6+语法
4. 组件按功能组织目录

## Git工作流

### 分支策略
- `main`: 主分支，保护分支，只接受合并请求
- `develop`: 开发分支，功能集成
- `feature/*`: 功能分支，从develop创建，合并回develop
- `release/*`: 发布分支，从develop创建，合并到develop和main
- `hotfix/*`: 热修复分支，从main创建，合并到develop和main

### 提交规范
```
<类型>(<范围>): <描述>

[详细描述]

[关联Issue]
```

类型：
- feat: 新功能
- fix: 修复bug
- docs: 文档更新
- style: 代码格式调整
- refactor: 代码重构
- test: 测试相关
- chore: 构建过程或辅助工具变动

## 数据库规范

### 命名规范
1. 表名使用小写字母加下划线：`user_info`
2. 字段名使用小写字母加下划线：`created_time`
3. 主键字段统一为：`id`
4. 外键字段：`关联表名_id`

### 索引规范
1. 主键索引：`pk_表名`
2. 唯一索引：`uk_表名_字段名`
3. 普通索引：`idx_表名_字段名`

## API设计规范

### RESTful API
- GET: 获取资源
- POST: 创建资源
- PUT: 更新资源
- DELETE: 删除资源
- PATCH: 部分更新资源

### 响应格式
```json
{
  "code": 200,
  "message": "成功",
  "data": {},
  "timestamp": 1646812800000
}
```

### 错误码
- 200: 成功
- 400: 请求参数错误
- 401: 未授权
- 403: 禁止访问
- 404: 资源不存在
- 500: 服务器内部错误

## 测试规范

### 单元测试
- 测试类名：`被测试类名Test`
- 测试方法名：`test方法名_场景_预期结果`
- 覆盖率要求：核心业务代码80%+

### 集成测试
- 测试API接口
- 测试数据库操作
- 测试外部服务集成

## 部署规范

### 环境划分
1. 开发环境 (dev)
2. 测试环境 (test)
3. 预发布环境 (staging)
4. 生产环境 (prod)

### 配置文件
- `application.yml`: 基础配置
- `application-dev.yml`: 开发环境配置
- `application-test.yml`: 测试环境配置
- `application-prod.yml`: 生产环境配置

## 安全规范

### 密码安全
1. 密码必须加密存储
2. 使用BCrypt加密算法
3. 密码复杂度要求：8位以上，包含大小写字母、数字、特殊字符

### 数据安全
1. 敏感数据加密存储
2. 数据传输使用HTTPS
3. SQL注入防护
4. XSS攻击防护

## 性能规范

### 数据库性能
1. 查询必须使用索引
2. 避免全表扫描
3. 大数据量分页查询
4. 定期清理历史数据

### 接口性能
1. 接口响应时间<2秒
2. 支持并发请求
3. 使用缓存减少数据库压力

## 文档规范

### 代码注释
1. 类注释：说明类的作用
2. 方法注释：说明方法的功能、参数、返回值
3. 复杂逻辑注释：说明业务逻辑

### API文档
1. 使用Swagger生成API文档
2. 文档必须及时更新
3. 提供示例请求和响应

## 代码审查

### 审查要点
1. 代码是否符合规范
2. 功能实现是否正确
3. 是否有潜在的性能问题
4. 是否有安全漏洞
5. 测试是否充分

### 审查流程
1. 创建Pull Request
2. 至少2人审查通过
3. 所有检查通过后才能合并
